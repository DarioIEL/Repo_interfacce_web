# Polimorfismo — Appunti Teorici

## Definizione

Il termine **polimorfismo** deriva dal greco *polys* (molti) e *morphé* (forma).  
In Java indica la capacità di un riferimento di tipo superclasse di assumere comportamenti diversi a seconda del tipo reale dell'oggetto a cui punta a runtime.

È uno dei quattro pilastri della programmazione orientata agli oggetti, insieme a **incapsulamento**, **ereditarietà** e **astrazione**.

---

## Tipi di Polimorfismo in Java

Java supporta due forme di polimorfismo:

| Tipo | Quando si risolve | Meccanismo |
|---|---|---|
| **Statico** (compile-time) | Durante la compilazione | Method overloading |
| **Dinamico** (runtime) | Durante l'esecuzione | Method overriding |

Nella OOP, quando si parla di polimorfismo senza ulteriori specifiche, si intende quasi sempre quello **dinamico**.

---

## Polimorfismo Statico — Overloading

Si ha overloading quando nella stessa classe esistono più metodi con lo **stesso nome ma firma diversa** (numero o tipo di parametri).

```java
public class Calcolatrice {
    public int somma(int a, int b) { return a + b; }
    public double somma(double a, double b) { return a + b; }
    public int somma(int a, int b, int c) { return a + b + c; }
}
```

Il compilatore decide quale metodo chiamare in base agli argomenti passati. Non richiede ereditarietà.

---

## Polimorfismo Dinamico — Overriding

Si ha overriding quando una sottoclasse **ridefinisce** un metodo già presente nella superclasse, mantenendo la stessa firma.

```java
public class Animale {
    public void emettiSuono() {
        System.out.println("Suono generico");
    }
}

public class Cane extends Animale {
    @Override
    public void emettiSuono() {
        System.out.println("Bau");
    }
}

public class Gatto extends Animale {
    @Override
    public void emettiSuono() {
        System.out.println("Miao");
    }
}
```

Il metodo da eseguire viene scelto a **runtime** in base al tipo concreto dell'oggetto, non al tipo del riferimento.

---

## Il Meccanismo: Dynamic Dispatch

Il **dynamic dispatch** (o late binding) è il meccanismo con cui la JVM risolve a runtime quale implementazione di un metodo eseguire.

```java
Animale a = new Cane(); // riferimento Animale, oggetto Cane
a.emettiSuono();        // stampa "Bau", non "Suono generico"
```

La JVM non guarda il tipo del riferimento (`Animale`), ma il tipo reale dell'oggetto in memoria (`Cane`). Questo accade per tutti i metodi non-statici e non-final.

> I metodi `static` e `final` non sono soggetti al dynamic dispatch: vengono risolti a compile-time.

---

## Upcasting

L'**upcasting** è l'assegnazione di un oggetto di una sottoclasse a un riferimento della superclasse.  
È sempre **implicito e sicuro**: ogni istanza di `Cane` è anche un `Animale`.

```java
Animale a = new Cane(); // upcasting implicito
```

Con un riferimento di tipo `Animale` puoi chiamare solo i metodi definiti in `Animale`, anche se l'oggetto reale è un `Cane`. I metodi aggiuntivi di `Cane` non sono accessibili direttamente.

---

## Downcasting

Il **downcasting** è l'operazione inversa: si riporta il riferimento al tipo concreto della sottoclasse.  
È sempre **esplicito** e comporta un rischio.

```java
Animale a = new Cane();

if (a instanceof Cane) {
    Cane c = (Cane) a; // downcasting esplicito
    c.recuperaOggetto(); // metodo specifico di Cane
}
```

Se si tenta un downcast su un tipo incompatibile senza controllo, la JVM lancia una `ClassCastException` a runtime.  
L'operatore `instanceof` va sempre usato prima di un downcast per verificare il tipo reale dell'oggetto.

> A partire da **Java 16**, il pattern matching semplifica la sintassi:
> ```java
> if (a instanceof Cane c) {
>     c.recuperaOggetto(); // c è già tipizzato, nessun cast esplicito
> }
> ```

## Classe Astratta vs Interfaccia

Entrambe sono strumenti per progettare gerarchie polimorfiche, ma con scopi diversi.

| | Classe Astratta | Interfaccia |
|---|---|---|
| Istanziabile | No | No |
| Metodi concreti | Sì | Sì (da Java 8, con `default`) |
| Attributi di istanza | Sì | No (solo costanti) |
| Ereditarietà multipla | No | Sì |
| Uso tipico | Condividere comportamento comune | Definire un contratto |

Una classe astratta modella una **relazione "è un"** (un `Cane` è un `Animale`).  
Un'interfaccia modella una **capacità** (un `Cane` può `Addestrabile`, `Adottabile`, ecc.).

---

## Perché il Polimorfismo è Importante

### Principio Open/Closed
Il codice che lavora su riferimenti della superclasse non deve essere modificato quando si aggiunge una nuova sottoclasse. Si estende il comportamento senza toccare il codice esistente.

### Riduzione dei condizionali
Senza polimorfismo si tende a scrivere lunghe catene di `if-else` o `switch` per gestire tipi diversi. Il polimorfismo elimina questa necessità delegando la decisione alla JVM.

### Sostituibilità (Liskov Substitution Principle)
Ogni sottoclasse deve poter sostituire la sua superclasse senza alterare il comportamento atteso del programma. Questo è il principio **L** dei principi SOLID.

---

## Errori Comuni

**1. Confondere il tipo del riferimento con il tipo dell'oggetto**  
Il tipo del riferimento determina *cosa puoi chiamare*. Il tipo dell'oggetto determina *quale implementazione viene eseguita*.

**2. Override involontario**  
Ridefinire un metodo cambiando solo il tipo di ritorno o modificando la firma produce overloading, non overriding. `@Override` serve proprio a fare in modo che il compilatore segnali questo errore.

**3. Downcast senza `instanceof`**  
Un downcast diretto senza verifica può compilare correttamente ma esplodere a runtime con `ClassCastException`.

**4. Metodi statici e polimorfismo**  
I metodi statici non partecipano al dynamic dispatch. Definire un metodo statico con lo stesso nome nella sottoclasse produce **method hiding**, non overriding — il metodo chiamato dipende dal tipo del *riferimento*, non dell'oggetto.

---

## Riepilogo

- Il polimorfismo dinamico si basa su **ereditarietà + overriding + upcasting**
- La JVM risolve il metodo corretto a **runtime** tramite dynamic dispatch
- L'upcasting è implicito; il downcasting è esplicito e richiede `instanceof`
- Classi astratte e interfacce sono i due strumenti principali per progettare gerarchie polimorfiche
- Il polimorfismo è alla base dei principi SOLID e dei principali framework Java
